# Directus flows reference

Last reviewed: 2026-07-09
Sources: `official-sources.md` -> Flow docs.

Use this for Flows module work, operation chains, automation troubleshooting, and flow logs.

## Concepts

- Flow = event-driven automation.
- Trigger = the event or manual/webhook/action that starts a flow.
- Operation = one action in the flow.
- Data chain = accumulated data passed between trigger and operations.
- `$last` = most recent operation output during execution.
- `$trigger` = trigger payload area.
- `$accountability` = permissions/request/user context generated by the trigger.
- `$env` = allowlisted environment variables available to the flow.
- Operation keys append operation output, or `null` when the operation generates no data.

## Operation hints

- Data operations can create, read, update, or delete collection items and can run with configured permission scope.
- Webhook / Request URL operations can call external URLs with custom method, headers, and body.
- Condition operations route success/reject paths; in blocking event hooks, a rejecting final condition can cancel the original database transaction.
- Directus Run Script operations are sandboxed JavaScript/TypeScript: no file-system access, no network requests, and no Node modules. This is separate from Pi `browser_run_script`.
- Flow inputs can often be switched to raw editor mode for direct JSON/data-chain variables.

## Why flows are high-risk

Flows can perform data changes, external calls, and arbitrary/elevated work depending on operations and accountability. Treat flow creation/editing as a high-risk mutation even when the UI looks simple.

## Flow fields to verify

Before saving a flow change, identify:

- Name and description.
- Status: active or inactive.
- Trigger type and trigger options.
- Operation sequence and each operation's key/name.
- Data-chain assumptions between operations.
- Whether the flow calls external services.
- Whether the flow changes data or schema.
- Whether the flow calls external URLs or sends notifications/email.
- Activity/log tracking setting.

## Logs

Flow logs can show trigger options, payload, accountability, and operation payloads. Logs may contain sensitive data. Summarize only what is needed and avoid copying private payloads into durable files.

## Safety gates

- Require explicit user intent for creating, editing, enabling, disabling, or deleting flows.
- Prefer leaving new/changed flows inactive until reviewed unless the user explicitly asks to activate.
- After a flow save, verify with UI state, network response, or a fresh read.
- Do not run manual flows against production-like data unless the user explicitly asks.
