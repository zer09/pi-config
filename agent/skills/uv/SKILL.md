# uv

Use uv for Python project, script, package, dependency, and tool workflows unless the repo clearly uses another manager.

## Routing

Prefer uv when you see:

- `uv.lock`
- `[project]` in `pyproject.toml` with uv-managed workflows
- requirements files generated by uv

Do not override an existing Poetry or PDM project unless the user asks to migrate:

- `poetry.lock` means use Poetry conventions.
- `pdm.lock` means use PDM conventions.

## Commands

### Scripts

```bash
uv run script.py
uv run --with requests script.py
uv add --script script.py requests
```

Use `uv run --with <package> python <script.py>` when a script needs dependencies that are not already installed.

### Projects

```bash
uv init
uv add requests
uv remove requests
uv sync
uv run <command>
uv run python -c "..."
uv run -p 3.12 <command>
```

### One-off tools

```bash
uvx ruff check .
uvx ty check
uvx <tool>@<version> <args>
```

Use `uvx` for well-known tools only. Use `uv tool install` only when the user explicitly asks for a persistent install.

### Legacy requirements workflows

```bash
uv venv
uv pip install -r requirements.txt
uv pip compile requirements.in -o requirements.txt
uv pip sync requirements.txt
uv pip compile --universal requirements.in -o requirements.txt
```

Use the pip interface only for legacy requirements workflows. Do not introduce new `requirements.txt` files when a uv project is appropriate.

## Local policy

- Prefer `uv run ...` over bare `python`, `python3`, or activating virtualenvs.
- Prefer `uv add` and `uv remove` over `pip install` in uv projects.
- Prefer `uvx <tool>` for one-off Python CLI tools.
- Keep changes scoped to the active project; do not migrate package managers without explicit instruction.
